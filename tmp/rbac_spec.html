1 Das Grundprinzip

Ein sauberes RBAC trennt zwei Dimensionen die heute in Decivio vermischt werden. Diese Trennung ist der Kern von allem.

Dimension 1 — Rollen (WER darf WAS tun)

Steuert Aktionen: Darf dieser Nutzer eine Entscheidung erstellen? Darf er ein Review abgeben? Darf er Nutzer einladen?

Gespeichert in: org_members.role

Dimension 2 — Plan (WELCHE Features sind verfügbar)

Steuert Features: Ist der Risk Register für diese Organisation freigeschaltet? Ist der Executive Hub buchbar?

Gespeichert in: organizations.plan

Die goldene Regel

Rollen und Plan sind unabhängig. Ein Viewer auf Enterprise-Plan sieht mehr Features als ein Viewer auf Starter — aber er kann trotzdem nichts bearbeiten. Ein Owner auf Starter-Plan kann alles verwalten — hat aber weniger Features zur Verfügung.

Nutzer macht Aktion
→
Rollen-Check: Darf er das?
→
Plan-Check: Ist Feature verfügbar?
→
Beide ✅ → Erlaubt

2 Die 6 Rollen

Jede Rolle hat eine klare Definition, einen typischen Nutzer und abgegrenzte Rechte.

OWNER

👑 Owner

Gründer, IT-Verantwortlicher, Org-Admin
  • Alles was Admin kann
  • Billing verwalten
  • Plan upgraden / downgraden
  • Org löschen oder umbenennen
  • Andere Owner ernennen
  • Kann nicht entfernt werden außer durch anderen Owner
ADMIN

⚙️ Admin

Abteilungsleiter, Operations Manager
  • Nutzer einladen und Rollen vergeben
  • Templates erstellen und bearbeiten
  • SLA-Regeln konfigurieren
  • Automationen verwalten
  • Alle Entscheidungen sehen und bearbeiten
  • Audit Trail einsehen
  • Teams erstellen und verwalten
EXECUTIVE

📊 Executive

CFO, CEO, COO, Vorstand
  • Alle Entscheidungen lesen (org-weit)
  • Executive Hub und Board Report
  • Risk Register lesen
  • Analytics und Insights
  • CEO Briefing
  • Keine Entscheidungen erstellen
  • Kein User-Management
MEMBER

🎯 Member

Projektleiter, Teamleiter, normaler Nutzer
  • Entscheidungen im eigenen Team erstellen
  • Reviews abgeben
  • Tasks erstellen und verwalten
  • Eigene Entscheidungen bearbeiten
  • Risiken erfassen (im eigenen Team)
  • Templates nutzen (nicht erstellen)
  • Nur eigenes Team sehen
REVIEWER

🔍 Reviewer

Externe Berater, Compliance, Legal
  • Zugewiesene Entscheidungen lesen
  • Reviews abgeben (Approve / Reject)
  • Kommentare schreiben
  • Keine eigenen Entscheidungen erstellen
  • Kein Zugriff auf Analytics
  • Kein Zugriff auf Risk Register
  • Sieht nur explizit zugewiesene Items
VIEWER

👁️ Viewer

Aufsichtsrat, externe Stakeholder
  • Nur lesen — gar keine Aktionen
  • Entscheidungen des eigenen Teams lesen
  • Kein Analytics-Zugriff
  • Kein Risk Register
  • Kein Audit Trail
  • Kein Kommentieren
  • Kein Schreiben irgendwo
Wichtige Regel: Rollen sind org-weit

Ein Nutzer hat eine Rolle für die gesamte Organisation. Team-spezifische Einschränkungen werden über team_members Tabelle zusätzlich geregelt — aber die Basis-Rechte kommen immer aus der Org-Rolle. Ein Member sieht standardmäßig nur sein Team — nicht weil er kein Admin ist, sondern weil RLS das steuert.

4 Datenbankstruktur

Drei neue / geänderte Tabellen. Der Rest der bestehenden 40+ Tabellen bleibt unverändert.

4.1 — org_members (neue zentrale Tabelle)

Ersetzt die bisherige team_members-Logik für Org-weite Rollen. Team-Mitgliedschaft bleibt separat.

-- Zentrale Rollen-Tabelle pro Organisation CREATE TABLE org_members ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), org_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE, user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE, role TEXT NOT NULL DEFAULT 'member', -- 'owner' | 'admin' | 'executive' | 'member' | 'reviewer' | 'viewer' invited_by UUID REFERENCES profiles(id), joined_at TIMESTAMPTZ DEFAULT now(), created_at TIMESTAMPTZ DEFAULT now(), UNIQUE(org_id, user_id), CHECK (role IN ('owner', 'admin', 'executive', 'member', 'reviewer', 'viewer')) );

4.2 — profiles (2 neue Felder)

-- Bestehende profiles Tabelle erweitern ALTER TABLE profiles ADD COLUMN view_mode TEXT DEFAULT 'default'; -- Steuert Dashboard-Ansicht: 'default' | 'executive' | 'admin' -- Wird beim ersten Login aus org_members.role abgeleitet ALTER TABLE profiles ADD COLUMN decision_count INTEGER DEFAULT 0; -- Für Progressive Disclosure — wird bei jeder neuen Entscheidung hochgezählt

4.3 — role_permissions (optional, für maximale Flexibilität)

Nur nötig wenn Enterprise-Kunden Custom-Rollen brauchen. Für den Start nicht nötig — Permissions sind hardcoded im Code.

-- Nur für Phase 2 / Enterprise Custom Roles CREATE TABLE role_permissions ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), org_id UUID REFERENCES organizations(id), -- null = globale Default-Permission role TEXT NOT NULL, permission TEXT NOT NULL, -- z.B. 'decisions.create', 'risk.read', 'analytics.view' enabled BOOLEAN DEFAULT true, UNIQUE(org_id, role, permission) );

4.4 — Hilfs-Funktion (wichtig für RLS)

-- Funktion die die Rolle des aktuellen Nutzers zurückgibt CREATE OR REPLACE FUNCTION get_user_role(p_org_id UUID) RETURNS TEXT AS $$ SELECT role FROM org_members WHERE user_id = auth.uid() AND org_id = p_org_id LIMIT 1; $$ LANGUAGE sql SECURITY DEFINER; -- Funktion die prüft ob Nutzer mindestens eine bestimmte Rolle hat CREATE OR REPLACE FUNCTION has_role(p_org_id UUID, p_min_role TEXT) RETURNS BOOLEAN AS $$ SELECT EXISTS ( SELECT 1 FROM org_members WHERE user_id = auth.uid() AND org_id = p_org_id AND role = ANY( CASE p_min_role WHEN 'viewer' THEN ARRAY['viewer','reviewer','member','executive','admin','owner'] WHEN 'member' THEN ARRAY['member','executive','admin','owner'] WHEN 'executive' THEN ARRAY['executive','admin','owner'] WHEN 'admin' THEN ARRAY['admin','owner'] WHEN 'owner' THEN ARRAY['owner'] END ) ); $$ LANGUAGE sql SECURITY DEFINER;

5 RLS-Policies (Supabase)

Row-Level Security stellt sicher dass die Rechte nicht nur im Frontend sondern direkt in der Datenbank durchgesetzt werden. Kein Frontend-Bypass möglich.

Decisions — Beispiel RLS

-- Lesen: Eigenes Team ODER Admin/Executive/Owner CREATE POLICY decisions_select ON decisions FOR SELECT USING ( team_id IN ( SELECT team_id FROM team_members WHERE user_id = auth.uid() ) OR has_role(org_id, 'executive') OR (confidential = false AND has_role(org_id, 'viewer')) ); -- Erstellen: Nur Member, Admin, Owner CREATE POLICY decisions_insert ON decisions FOR INSERT WITH CHECK ( has_role(org_id, 'member') AND team_id IN ( SELECT team_id FROM team_members WHERE user_id = auth.uid() ) ); -- Bearbeiten: Eigene Entscheidung (Member) ODER Admin/Owner CREATE POLICY decisions_update ON decisions FOR UPDATE USING ( (owner_id = auth.uid() AND has_role(org_id, 'member')) OR has_role(org_id, 'admin') );

Analytics — nur für Executive+

-- Briefings: Nur Executive, Admin, Owner CREATE POLICY briefings_select ON briefings FOR SELECT USING ( has_role(org_id, 'executive') OR user_id = auth.uid() ); -- Risks: Lesen für Member (eigenes Team), Executive/Admin/Owner (alle) CREATE POLICY risks_select ON risks FOR SELECT USING ( (has_role(org_id, 'member') AND team_id IN ( SELECT team_id FROM team_members WHERE user_id = auth.uid() )) OR has_role(org_id, 'executive') );

6 Frontend-Logik

Ein zentraler Hook der alles steuert. Alle Komponenten fragen nur diesen Hook — nie direkt die Datenbank.

usePermissions Hook (neu)

// hooks/usePermissions.ts const usePermissions = () => { const { profile } = useAuth(); const { orgMember } = useOrgMember(); const role = orgMember?.role ?? 'viewer'; const can = { // Entscheidungen createDecision: ['owner','admin','member'].includes(role), editAnyDecision: ['owner','admin'].includes(role), deleteDecision: ['owner','admin'].includes(role), // Reviews submitReview: ['owner','admin','member','reviewer'].includes(role), assignReviewer: ['owner','admin','member'].includes(role), // Analytics viewAnalytics: ['owner','admin','executive'].includes(role), viewExecutiveHub:['owner','admin','executive'].includes(role), // Administration manageUsers: ['owner','admin'].includes(role), manageTemplates: ['owner','admin'].includes(role), manageBilling: role === 'owner', }; return { role, can }; };

8 Progressive Disclosure Integration

RBAC und Progressive Disclosure arbeiten zusammen — aber bleiben getrennte Systeme.

Die Kombinationslogik

Ein Feature ist sichtbar wenn DREI Bedingungen erfüllt sind:

1. Rolle erlaubt es — has_role('executive') z.B.

2. Plan enthält es — features.analyticsHub = true

3. Progressive Disclosure freigegeben — decision_count >= 10 ODER override = true

// Vollständige Sichtbarkeits-Logik const isVisible = (feature: string) => { const { can } = usePermissions(); const { features } = useFeatureFlags(); const { decision_count, progressive_override } = useProfile(); // Override — alles sofort zeigen (für Demo / Power User) if (progressive_override) { return can[feature] && features[feature]; } // Stufe 1 Features — immer sofort const stage1 = ['dashboard', 'decisions', 'tasks', 'calendar']; if (stage1.includes(feature)) return can[feature]; // Stufe 2 Features — nach 10 Entscheidungen const stage2 = ['riskRegister', 'templates', 'auditTrail', 'automations']; if (stage2.includes(feature)) { return can[feature] && features[feature] && decision_count >= 10; } // Stufe 3 Features — nach 25 Entscheidungen return can[feature] && features[feature] && decision_count >= 25; };

10 Was jetzt, was später

🟢 Jetzt — Phase 1 (15–20h)

org_members Tabelle, has_role() Funktion, usePermissions Hook, 3 Dashboard-Views, Onboarding-Rollen-Screen, Admin-Rollen-UI, Progressive Override. Damit kannst du die ersten 50 Kunden problemlos bedienen.

🔵 Nach ersten 10 Kunden — Phase 2

Alle RLS-Policies auf neues System umstellen, role_permissions für Custom Roles, granulares Logging wer was wann gesehen hat. Kosten: 10–15 weitere Stunden.

🟣 Enterprise-Deals — Phase 3

SSO (SAML/OIDC), Active Directory Integration, Custom Roles per Organisation, granulare Permission-Overrides, Compliance-Reporting wer Zugriff auf was hat. Das baust du wenn ein konkreter Enterprise-Kunde €499+/Monat zahlt.

Das Wichtigste

Die Datenstruktur die du heute mit org_members.role anlegst ist kompatibel mit allen drei Phasen. Du machst jetzt keine Entscheidung die du später bereust — du legst das Fundament auf dem alles aufbaut.
